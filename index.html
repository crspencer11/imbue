Part III

Tutorial: Vet as a pre-commit hook

Use case: Catching AI mistakes before they leave your machine

The docs show Vet in CI and as a CLI. This tutorial covers a use case that isn't documented: running Vet automatically as a pre-commit hook — so every time you commit AI-generated code, Vet quietly checks the diff before the commit lands. You get CI-quality review feedback in 10 seconds, before the code ever reaches your team.

Why this use case matters

The GitHub Action setup is great for catching things in PRs — but by then, your teammates have already seen the code, CI has already run, and reverting is socially awkward. A pre-commit hook short-circuits that: it's your own private code reviewer that catches agent hallucinations, logic errors, and security issues while you still think of the code as "in progress."

This is especially useful if you're using Sculptor to run multiple agents and manually merging their output — you want a sanity check before that merged code becomes a commit.

Prerequisites

Python 3.10+, pip, an Anthropic API key (or another supported provider), and a git repo you actually care about.

1

Install Vet

Vet is on PyPI as verify-everything

# Create virtual env
python3 -m venv .venv
source .venv/bin/activate
        
pip install verify-everything
        
# Verify the install
vet --version

Then set your API key — Vet uses Anthropic by default:

export ANTHROPIC_API_KEY=sk-ant-...
# Add to ~/.zshrc or ~/.bashrc to persist
2

Write the pre-commit hook script

In your repo root, create .git/hooks/pre-commit:

#!/bin/bash
# Vet pre-commit hook
# Runs vet against your staged changes before committing.

# Only run if vet is installed
if ! command -v vet &> /dev/null; then
  echo "[pre-commit] vet not found — skipping AI review"
  exit 0
fi

# Only run if ANTHROPIC_API_KEY is set (or another provider key)
if [ -z "$ANTHROPIC_API_KEY" ]; then
  echo "[pre-commit] No API key — skipping vet"
  exit 0
fi

echo "[pre-commit] Running vet on staged diff..."

# Run vet against staged changes vs HEAD
vet "Review staged changes for correctness" --base-commit HEAD

EXIT_CODE=$?

# Exit code 10 = issues found; exit code 0 = clean
if [ $EXIT_CODE -eq 10 ]; then
  echo "[pre-commit] vet found issues. Commit aborted."
  echo "Fix the issues above, or run 'git commit --no-verify' to skip."
  exit 1
fi

exit 0

Make it executable:

chmod +x .git/hooks/pre-commit
3

Try it on real AI-generated code

Stage some code you got from Claude Code or Codex and commit. You'll see something like:

$ git commit -m "feat: add payment processing"
[pre-commit] Running vet on staged diff...

Issues found:

1. [insecure_code] payment.py:47
   User-supplied card number is logged in plain text before being
   passed to the payment processor. This would expose sensitive data
   in log files.

2. [logic_error] payment.py:83
   Refund amount is calculated before checking whether the original
   transaction exists. If the transaction lookup fails, the refund
   amount defaults to 0 and no error is raised.

[pre-commit] vet found issues. Commit aborted.
Fix the issues above, or run 'git commit --no-verify' to skip.
4

Customize what vet checks (optional)

Create a .vet/configs.toml in your repo root to set a profile for pre-commit usage specifically — faster, focused on security and logic only:

[profiles.precommit]
model = "claude-sonnet-4-20250514"
issue_codes = ["logic_error", "insecure_code", "unverified_claim"]
output_format = "text"
severity_threshold = "medium"

Then update the hook to use the profile:

vet "Review staged changes" --base-commit HEAD --profile precommit
5

Share the hook with your team (the right way)

Git hooks aren't tracked by git by default. If you want your team to use this hook, put it in a .githooks/ directory and configure git:

# Put the hook here instead:
mkdir -p .githooks
cp .git/hooks/pre-commit .githooks/pre-commit

# Tell git to use this directory:
git config core.hooksPath .githooks

# Commit .githooks/ to your repo
git add .githooks && git commit -m "chore: add vet pre-commit hook"
NoteThe hook will only run for teammates who have vet installed and an API key set. The script fails gracefully if either is missing, so it won't block anyone who hasn't opted in.

Issues & friction I found during setup

  • Exit code behavior underdocumented — I had to read the source to find that exit code 10 = issues found. This is critical for scripting. It should be in a "Scripting & automation" section of the README.
  • --base-commit for staged changes — Using HEAD to diff staged changes is a bit counterintuitive; I expected something like --staged. Would benefit from a "Common recipes" section covering pre-commit, pre-push, and CI.
  • First run is slow — The initial LLM call takes 15–30 seconds on a medium diff. Fine for CI, noticeable in a pre-commit hook. Docs could mention that --profile with fewer issue codes speeds this up significantly.

Who should use this

Solo devs shipping fast Moving too fast for careful review — want a backstop before code leaves their machine
Teams using AI agents heavily Sculptor users who merge output from multiple agents and want pre-merge sanity checks
Security-conscious engineers Fintech, healthcare, any regulated domain where AI-generated code needs extra scrutiny
Tech leads reviewing PRs Want to run vet before they review so they're not manually catching what a tool could catch
The bigger picture This tutorial represents a workflow where Vet, Sculptor, and Offload work in sequence: Sculptor runs parallel agents, you merge what looks good, Vet reviews the merged output as a pre-commit hook, and Offload runs your full test suite in isolated sandboxes before the PR opens. That's a real, working agentic development loop — and it's worth spelling out explicitly in Imbue's docs.
